# 호스트 운영 대시보드 MVP PRD

## 🎯 핵심 정보

**목적**: 숙소 호스트가 예약, 메시지, 숙소 상태, 운영 성과를 한 화면 체계에서 빠르게 관리하도록 돕는다.
**사용자**: 여러 숙소 또는 단일 숙소를 직접 운영하며 일일 예약과 게스트 응대를 처리하는 숙소 호스트.

## 🚶 사용자 여정

1. 로그인 페이지
   ↓ 이메일과 비밀번호 입력

2. 인증 확인
   ↓
   로그인 성공 → 대시보드 홈
   로그인 실패 → 로그인 페이지에서 오류 표시
   계정 없음 → 회원가입 페이지

3. 대시보드 홈
   ↓ 사용자가 처리할 업무 선택
   예약 요청 확인 → 예약 관리 페이지
   읽지 않은 메시지 확인 → 메시지 페이지
   숙소 운영 상태 확인 → 숙소 관리 페이지
   운영 성과 확인 → 성과 페이지

4. 업무 처리
   ↓
   예약 승인/거절 완료 → 예약 관리 페이지
   메시지 답장 완료 → 메시지 페이지
   숙소 상태 변경 완료 → 숙소 관리 페이지
   성과 확인 완료 → 대시보드 홈 또는 다른 메뉴

## 🗄️ 데이터 모델

### Host

| 필드                | 설명                                | 타입/관계 |
| ------------------- | ----------------------------------- | --------- |
| id                  | 호스트 고유 식별자                  | UUID      |
| name                | 호스트 이름                         | string    |
| email               | 로그인 이메일                       | string    |
| responseTimeMinutes | **전체 평균 응답 시간 (상시 누적)** | number    |

### Listing

| 필드          | 설명             | 타입/관계 |
| ------------- | ---------------- | --------- |
| id            | 숙소 고유 식별자 | UUID      |
| hostId        | 숙소 소유 호스트 | → Host.id |
| title         | 숙소 이름        | string    |
| status        | 숙소 운영 상태   | string    |
| isPublic      | 공개 상태 토글   | boolean   |
| nightlyPrice  | 1박 가격         | number    |
| address       | 숙소 주소        | string    |
| coverImageUrl | 대표 이미지 URL  | string    |

### Reservation

| 필드        | 설명             | 타입/관계    |
| ----------- | ---------------- | ------------ |
| id          | 예약 고유 식별자 | UUID         |
| listingId   | 예약된 숙소      | → Listing.id |
| guestName   | 게스트 이름      | string       |
| status      | 예약 상태        | string       |
| totalAmount | 예약 총액        | number       |
| checkIn     | 체크인 날짜      | Date         |
| checkOut    | 체크아웃 날짜    | Date         |
| guestCount  | 게스트 인원 수   | number       |
| createdAt   | 예약 생성 일시   | Date         |

### MessageThread

| 필드          | 설명                | 타입/관계        |
| ------------- | ------------------- | ---------------- |
| id            | 대화 고유 식별자    | UUID             |
| reservationId | 연결 예약           | → Reservation.id |
| guestName     | 게스트 이름         | string           |
| lastMessage   | 최근 메시지         | string           |
| unreadCount   | 읽지 않은 메시지 수 | number           |
| status        | 대화 상태           | string           |

### Message

| 필드     | 설명               | 타입/관계          |
| -------- | ------------------ | ------------------ |
| id       | 메시지 고유 식별자 | UUID               |
| threadId | 소속 대화          | → MessageThread.id |
| sender   | 발신자             | "host" \| "guest"  |
| body     | 메시지 내용        | string             |
| sentAt   | 발송 일시          | Date               |
| isRead   | 읽음 여부          | boolean            |

### PerformanceSummary

| 필드                | 설명                                | 타입/관계              |
| ------------------- | ----------------------------------- | ---------------------- |
| id                  | 성과 요약 식별자                    | UUID                   |
| listingId           | 대상 숙소                           | → Listing.id           |
| revenue             | 매출                                | number                 |
| bookingCount        | 예약 수                             | number                 |
| occupancyRate       | 점유율                              | number                 |
| period              | 집계 기간                           | "7d" \| "30d" \| "90d" |
| responseTimeMinutes | **선택 기간 내 평균 응답 시간(분)** | number                 |

**응답 시간 필드 분리 원칙**: 대시보드 홈은 `Host.responseTimeMinutes`(상시 평균)를, 성과 페이지는 `PerformanceSummary.responseTimeMinutes`(선택 기간 평균)를 표시한다.

### Status 정의

- `Reservation.status`: `"pending" | "confirmed" | "rejected" | "cancelled" | "completed"`
- `Listing.status`: `"active" | "inactive" | "maintenance"`
- `Listing.isPublic`: `boolean` — 공개 상태 토글용 별도 필드
- `MessageThread.status`: `"unread" | "read" | "archived"`

## 인증 구현 방식

- **방식**: 쿠키 기반 목업 세션 (백엔드 없음)
- **구현**: Next.js Server Action에서 로그인 처리 → httpOnly 쿠키에 호스트 ID 저장
- **보호**: `middleware.ts`의 `matcher`를 `["/dashboard/:path*"]`로 설정. 모든 인증 필요 페이지는 `/dashboard` 하위로 통일하여 보호 누락 방지. 쿠키 없으면 `/login` 리다이렉트.
- **로그아웃**: Server Action에서 쿠키 삭제 후 `/login` 리다이렉트
- **목업 사용자**: 이메일/비밀번호 하드코딩된 배열로 검증 (실제 DB 불필요)
- **회원가입 처리**: Server Action에서 입력값(이름/이메일/비밀번호)을 메모리 모듈의 사용자 배열에 push한 뒤, 즉시 로그인 쿠키를 발급하고 `/dashboard`로 리다이렉트한다. 서버 재시작 시 추가된 사용자는 초기화된다(MVP 한정). 단, **동일 이메일이 이미 사용자 배열에 존재하면 가입을 거절하고 폼에 `이미 가입된 이메일입니다` 오류를 표시**한다.
- **데이터 격리 규칙**: 모든 목록 조회(예약, 메시지, 숙소, 성과)는 쿠키에 저장된 `hostId`로 1차 필터링한다. `Listing.hostId === currentHostId` 조건을 만족하는 데이터만 표시하며, 다른 호스트의 데이터는 응답에 포함되지 않는다. **적용 레이어**: 목업 데이터 모듈의 모든 조회 함수는 `hostId`를 필수 파라미터로 받아 내부에서 일치 항목만 반환한다. (페이지/Server Action에서 hostId를 명시적으로 전달)
